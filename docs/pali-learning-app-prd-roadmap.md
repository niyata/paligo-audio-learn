## หลักการตรวจและสอน (Domain Logic — บังคับสำหรับ AI Examiner)

ส่วนนี้เป็น **กฎวิชาการ** ที่ฟีเจอร์ Mock Exam, Translation Assistant, และระบบตรวจข้อสอบ (รวม Paligo web) ต้องยึด — ไม่ใช่แค่คะแนน stamp แบบง่าย

### 1. อัธยาหาร (Adhyahara) — หัวใจ ป.ธ. ๓ / สัมพันธ์ไทย

ข้อความบาลีมัก **ละคำเชื่อม** (Ellipsis) ผู้เรียนต้อง **เติมคำที่บัง implied** — กระบวนการนี้เรียก **อัธยาหาร**

- คำที่**ไม่มีในมคธดิบ** แต่ผู้เรียนเติมตามหลัก (เช่น ประธานที่สอดคล้องวาจก/ลิงค์/วจนะ, คำเปิด `"` เช่น วตฺวา คู่ อิติ) → **ถือว่าถูกต้อง** ห้ามตัดคะแนนว่า «สมมติคำ» หรือ hallucination
- ตรวจว่าคำที่เติม **สอดคล้อง** วาจก (กิริย/กิริยาเสม), ลิงค์, วจนะ ของประโยค
- UI/AI ต้องแยกแสดง: **คำในมคธ** vs **คำอัธยาหารที่ผู้เรียนเติม**

### 2. ระบบ ๓ ให้ (ศ / ส / ป)

**Source of truth:** [`docs/pali-ai/SCORING-3-HAI.md`](pali-ai/SCORING-3-HAI.md) · APPROVER agent

| รหัส | ความหมาย | เก็บคะแนน |
|------|----------|-----------|
| **ศ** | ผิดศัพท์ | ๑ |
| **ส** | ผิดสัมพันธ์ | ๒ |
| **ป** | ผิดประโยค | ๖ / ๑๒ / ๑๘ ตามบรรทัด |

แปลงรวมเป็น **๓ / ๒ / ๑ / ๐ ให้** · phase gate: วิชาแปลต้อง ≥ **๒ ให้** (RULE_04)

> Canonical spec: [`docs/pali-ai/SCORING-3-HAI.md`](pali-ai/SCORING-3-HAI.md) · [`docs/ai-examiner-system-prompt.md`](ai-examiner-system-prompt.md)

### 3. สถาปัตยกรรมแปลสองชั้น (Dual Translation)

ทุกคำแปลไทยต้องแยกเก็บ/แสดง **สองฟิลด์**:

| ฟิลด์ | ชื่อ | ใช้เมื่อ |
|-------|------|----------|
| `translationLiteral` | **แปลโดยพยัญชนะ** | รักษาเครื่องหมายไวยากรณ์ (อันว่า… สู่… ย่อม…) — **บังคับ ป.ธ. ๑–๓** |
| `translationIdiomatic` | **แปลโดยอรรถ** | ภาษาไทยลื่น อ่านเป็นงานวรรณ — **เน้น ป.ธ. ๔–๙** |

- ป.ธ. ๑–๓: แสดงพยัญชนะเป็นหลัก · อรรถเป็นเสริม/เฉลย
- ป.ธ. ๔–๙: แสดงทั้งคู่ · ตรวจตามประเภทข้อสอบ

### 4. แนวทางสอน (Pedagogical Agent)

AI / ติวเตอร์ในแอป **ห้ามให้เฉลยทันที** — ใช้ scaffolding:

- ชี้ **รูปแบบที่คล้าย** (pattern recognition)
- **ไฮไลต์** ปัจจัย/วิภัตติ (color-code stem + suffix)
- อธิบาย **ทำไม** คำสองคำเกี่ยวกัน ก่อนตั้งชื่อสัมพันธ์ทางการ
- คำใบ้แบบขั้นบันได — ไม่เปิดเฉลยเต็มจนกว่าผู้เรียนจะลองแล้ว

## Pain Points ของผู้เรียน

- ต้องท่องจำศัพท์ ธาตุ วิภัตติ และกฎไวยากรณ์จำนวนมาก
- หนังสือตารางแจกศัพท์และตำราอ้างอิงใช้งานบนมือถือไม่สะดวก
- การแยกโครงสร้างประโยคบาลีทำได้ยาก โดยเฉพาะ **สัมพันธ์ไทย** และการเติม **อัธยาหาร**
- แอป/AI ทั่วไปมัก penalize คำที่ผู้เรียนเติมถูกต้อง หรือให้เฉลยเร็วเกินไป
- การค้นคำบาลีมักต้องรู้รูปศัพท์ตั้งต้นก่อน จึงค้นได้แม่น
- ข้อสอบเก่าและเฉลยกระจายอยู่หลายแหล่ง
- ผู้เรียนขาดระบบติดตามความก้าวหน้าและการทบทวนแบบเป็นรอบ
- **(Paligo Web)** อยากเรียนและส่งข้อสอบตรวจได้ แต่ **ยังไม่มีอาจารย์ในระบบ** — ดู [`prd-teacher-matching-onboarding.md`](prd-teacher-matching-onboarding.md)



## ฟีเจอร์หลัก



### 1. Interactive Declension & Conjugation Tables

ตารางแจกวิภัตติและการผันกริยาแบบ Interactive สำหรับนามศัพท์และอาขยาต

ความสามารถหลัก:

- ค้นหรือเลือกศัพท์ตัวอย่าง เช่น ปุริส, มุนิ
- แสดงรูปแจกตามวิภัตติ ลิงค์ และวจนะ
- แสดงการผันกริยาอาขยาตตามกาล บุรุษ และวจนะ
- รองรับการอธิบายที่มาของรูปศัพท์แบบสั้น

ประโยชน์:

- ลดการพึ่งหนังสือตารางหนา ๆ
- ช่วยผู้เรียนเห็น pattern การเปลี่ยนรูปได้เร็วขึ้น



### 2. Smart Pali-Thai Dictionary

พจนานุกรมบาลี-ไทยแบบ Offline พร้อมระบบค้นหาที่เหมาะกับอักษรบาลีในบริบทภาษาไทย

ความสามารถหลัก:

- ค้นคำบาลีและคำแปลไทยแบบรวดเร็ว
- รองรับอักขระบาลี เช่น นิคหิต พินทุ ฐ ญ และรูปสะกดที่พบบ่อย
- วิเคราะห์รากศัพท์เบื้องต้น เช่น ธาตุ ปัจจัย วิภัตติ
- ตัวอย่าง: ค้นคำว่า "คจฺฉติ" แล้วแสดงว่าเกี่ยวกับ "คมฺ" ธาตุ พร้อมรูปประกอบ

ประโยชน์:

- ผู้เรียนไม่จำเป็นต้องรู้รูปตั้งต้นของคำทั้งหมดก่อนค้น
- ใช้งานได้แม้ไม่มีอินเทอร์เน็ต



### 3. Flashcards & Spaced Repetition

ระบบทบทวนความจำแบบเป็นรอบ สำหรับศัพท์ ธาตุ กฎไวยากรณ์ และอัพยยศัพท์

ความสามารถหลัก:

- สร้างชุดการ์ดตามหมวด เช่น อัพยยศัพท์ ธาตุ สนธิ สมาส
- ใช้หลักการ spaced repetition คล้าย Anki
- บันทึกสถานะความจำและความคืบหน้าของผู้เรียน
- รองรับการ์ดแบบพลิกหน้า-หลังด้วย SwiftUI Animation

ประโยชน์:

- ลดการท่องจำแบบกระจัดกระจาย
- ช่วยให้ผู้เรียนทบทวนก่อนลืม



### 4. Interactive Translation Assistant

หน้าจออ่านและฝึกแปล/สัมพันธ์ ตาม **corpus ตามชั้น** (ดูตาราง ป.ธ.) — ไม่ใช่ธัมมปทัฏฐกถาทั้ง ๘ เล่มในทุกชั้น

ความสามารถหลัก:

- แสดงเนื้อหาจากตำราที่ถูกชั้น (เช่น ป.ธ. ๑–๒ → ภาค ๑–๔ · ป.ธ. ๓ → ภาค ๕–๘ + โหมดสัมพันธ์ไทย)
- แตะคำบาลีเพื่อดูคำแปล **สองชั้น**: แปลโดยพยัญชนะ / แปลโดยอรรถ (ตาม § Dual Translation)
- โหมด ป.ธ. ๓: รองรับ **อัธยาหาร** — ผู้เรียนเติมคำ implied ได้ ระบบไม่ penalize ถ้าสอดคล้องวาจก·ลิงค์·วจนะ
- แสดงบทบาทของคำ (ประธาน กริยา ส่วนขยาย กรรม) + **color-code** ปัจจัย/วิภัตติ
- คำใบ้แบบ **scaffolded** (pattern → ทำไม → ชื่อสัมพันธ์) ไม่เฉลยทันที

ประโยชน์:

- สอดคล้องหลักสูตรสนามหลวงจริง แยกชั้นและตำรา
- ฝึกสัมพันธ์ไทยและแปลสองชั้นตามมาตรฐานแม่กอง



### 5. Mock Exams & Timer

คลังข้อสอบเก่าสนามหลวงพร้อมระบบจับเวลา — ตรวจคำตอบด้วย **ระบบหน (Hana)** และ AI Examiner ตาม domain logic

ความสามารถหลัก:

- รวมข้อสอบเก่า ป.ธ. ๑–๙ แยก **วิชา + corpus ตามชั้น** (ไม่ปนเล่มตำราข้ามชั้น)
- ฝึกทำข้อสอบพร้อมตัวจับเวลา
- ตรวจแปล/สัมพันธ์: หัก ๖/๒/๑ หน · สรุป **สอบตก** เมื่อเกิน ๑๒ หน หรือ ๒ ไห
- รองรับอัธยาหารที่ถูกต้องในโหมดสัมพันธ์ไทย
- บันทึกผล ประวัติหักหน และเฉลย (พยัญชนะ/อรรถ ตามชั้น)
- AI feedback แบบ pedagogical (คำใบ้ก่อนเฉลย)

ประโยชน์:

- ซ้อมสอบด้วยเกณฑ์เดียวกับข้อสอบจริงสนามหลวง
- ไม่ให้คะแนนผิดเพราะ penalize คำอัธยาหารที่ถูกต้อง

### Phase 1: Foundation & Dictionary

ระยะเวลาโดยประมาณ: เดือนที่ 1

งานหลัก:

- สร้างโครงสร้างแอปด้วย SwiftUI

ผลลัพธ์ที่คาดหวัง:

- แอปมีโครงสร้างข้อมูลพร้อมต่อยอดฟีเจอร์อื่น



### Phase 2: Memorization Tool

ระยะเวลาโดยประมาณ: เดือนที่ 2

งานหลัก:

ผลลัพธ์ที่คาดหวัง:

- แอปเริ่มเป็นเครื่องมือเรียนประจำวันได้



### Phase 3: Translation Assistant & Content

ระยะเวลาโดยประมาณ: เดือนที่ 3-4

งานหลัก:

ผลลัพธ์ที่คาดหวัง:

- ป.ธ. ๓ ฝึกสัมพันธ์ไทยและอัธยาหารได้ในแอป

### Content Model ที่ควรออกแบบ

- `PaliTerm`: คำบาลี, คำแปล, หมวด, รากศัพท์, metadata
- `GrammarTopic`: หัวข้อไวยากรณ์, เนื้อหา, ตัวอย่าง
- `Flashcard`: หน้าแรก, หลังการ์ด, หมวด, รอบทบทวน
- `TextPassage`: ข้อความบาลี, แหล่งที่มา, **grade (ป.ธ.)**, **corpusId** (เล่ม/ภาค)
- `TokenAnnotation`: คำใน passage, ความหมาย, หน้าที่ทางไวยากรณ์, ลิงก์ไปยังศัพท์
- `TranslationPair`: `translationLiteral`, `translationIdiomatic` (dual translation)
- `AdhyaharaToken`: คำที่ผู้เรียนเติม, ไม่มีในมคธดิบ, เกณฑ์วาจก/ลิงค์/วจนะ, ถูก/ผิด
- `HanaDeduction`: ระดับ (major/medium/minor), หน (6/2/1), เหตุผล, รวมหน, สถานะสอบตก
- `ExamPaper`: ปี, ชั้นประโยค, วิชา, corpus, ข้อสอบ, เฉลย (พยัญชนะ/อรรถ)

## MVP Scope

MVP ควรเน้นฟีเจอร์ที่ทำให้ผู้เรียนใช้งานได้จริงตั้งแต่วันแรก

- พจนานุกรมบาลี-ไทย Offline
- บทเรียนบาลีไวยากรณ์พื้นฐาน
- Flashcards สำหรับศัพท์และอัพยยศัพท์
- ตารางแจกวิภัตติพื้นฐาน
- หน้าฝึกอ่านธัมมปทัฏฐกถา **ภาค ๑–๔** (ป.ธ. ๑–๒) แบบแตะคำศัพท์ + แปลพยัญชนะ



## ความเสี่ยงและข้อควรระวัง

- คุณภาพฐานข้อมูลศัพท์เป็นหัวใจสำคัญ หากข้อมูลไม่แม่น แอปจะเสียความน่าเชื่อถือ
- **Corpus ผิดชั้น** (เช่น ใช้ธัมมปทัฏฐกถา ๘ เล่มใน ป.ธ. ๑–๒) ทำลายความน่าเชื่อถือทันที — ต้องผูก passage/exam กับ `grade` + `corpusId`
- **Penalize อัธยาหารที่ถูก** ในโหมดสัมพันธ์ไทย = bug ร้ายแรง ไม่ใช่ UX เล็กน้อย
- **ใช้ rubric ทั่วไปแทน Hana** ทำให้คะแนนไม่สอดคล้องข้อสอบจริง
- เนื้อหาตำราและข้อสอบควรตรวจสอบสิทธิ์การใช้งานและแหล่งอ้างอิง
- การวิเคราะห์รากศัพท์อัตโนมัติอาจซับซ้อน ควรเริ่มจาก rule-based และ curated data ก่อน
- การจัดรูปแบบอักษรบาลีต้องทดสอบบนอุปกรณ์จริงหลายขนาด
- ข้อสอบและเฉลยต้องแยก metadata ให้ชัดเจน เช่น ปี วิชา ชั้น corpus และแหล่งที่มา
- AI Examiner ต้อง load domain rules (อ้างอิง `docs/ai-examiner-system-prompt.md`) — prompt ทั่วไปให้เฉลยเร็วเกินไป

## สรุป

แอปเรียนภาษาบาลีสำหรับบริบทไทยต้องยึด **หลักสูตรบาลีสนามหลวง** อย่างเคร่งครัด: corpus ตามชั้น (ป.ธ. ๑–๙), อัธยาหารใน สัมพันธ์ไทย, ระบบหน, แปลสองชั้น (พยัญชนะ/อรรถ), และการสอนแบบ scaffold — ไม่ใช่แค่พจนานุกรมและ flashcard ทั่วไป

จุดแข็งควรอยู่ที่ความถูกต้องตามแม่กองบาลี AI Examiner ที่ไม่ penalize อัธยาหารที่ถูก, และเครื่องมือฝึกที่สอดคล้องข้อสอบจริง
